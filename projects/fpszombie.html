Overview

FPS Zombie is a first-person zombie survival game where players enter a town-based environment, collect weapons, and defend themselves against zombies that aggressively track the nearest player. The core objective is simple: survive as long as possible.

The project includes both single-player and multiplayer flows, along with gameplay systems for movement, shooting, pickups, health, death, UI, audio feedback, and networked player support.

  • First-person gameplay programming in Unity using C#
  • Combat and weapon interaction systems
  • Enemy pursuit and survival loop design
  • HUD and player feedback systems
  • Multiplayer architecture using Unity Netcode for GameObjects
  • Lobby and Relay integration for session creation and joining

What I Worked On

Network Camera Positioner

I implemented the networked camera positioning setup for player-owned characters in the multiplayer scene.

  • Setup -The camera and audio listener are attached to the player prefab but toggled off by default so they don't activate for every connected client.
  • Turn On - When the network object spawns, a check runs to see if the local client is the owner. If so, the camera and audio listener are enabled - no other player activates them.
  • Update - Every Update(), if the local client owns this object, the camera position and rotation are continuously synced to the player's transform to stay locked to their view.

This makes sure each player only controls and hears from their own perspective in multiplayer, which is a core requirement for a networked FPS.

Network Camera Controller

I built the owner-only camera controller that handles mouse input and cursor state for the local player.

  • Setup - The script is attached directly to the camera object on the player prefab.
  • Start - On startup, if the local client owns this object, the cursor is locked to the center of the screen and hidden - standard FPS behavior.
  • Update - Only runs for the owning client. If the game is paused, the cursor unlocks and becomes visible for menu interaction. Otherwise, it reads Mouse X and Mouse Y axis values and uses them to rotate the camera for first-person look.

Wrapping all input in an owner check means remote player objects don't read local mouse input - a critical separation in multiplayer where every player object exists in the scene simultaneously.

Player Data

I created a shared player data system to keep player-specific information consistent across the network.

  • Stores data that needs to be visible and consistent for all connected players, not just the local one.
  • Acts as an extensible location for player identity and session-level state.
  • Can be used to enforce entry rules - for example, checking if an incoming player shares a name with an existing player and blocking them from joining the lobby if so.

Having a centralized, networked data container prevents systems from needing to reach into individual player objects to grab shared info. It also makes lobby validation and future multiplayer features easier to build on top of.

Zombie Game Multiplayer

I built the main multiplayer session manager using Netcode for GameObjects that persists with the host across scenes.

  • Setup - Attached to an empty object in the scene where the server and lobby are initialized. Marked as DontDestroyOnLoad so it carries forward with the host through scene transitions.
  • Single Player - When the player clicks the single-player button on the main menu, a boolean is set to false. This allows the game to start immediately as a host and then load the map.
  • Multiplayer - When the player clicks the multiplayer button, the boolean is set to true. This bypasses starting a server automatically and instead loads into the lobby creation scene.
  • Other - This class also handles join validation checks, initializes PlayerData, and sets up server-related delegates used by other systems throughout the project.

Centralizing the single-player vs. multiplayer branching logic into one persistent manager keeps the flow clean and predictable. Other systems can rely on this object being present and initialized rather than managing their own startup paths.

Zombie Game Lobby

I built the full lobby system using Unity's Lobby and Relay packages, supporting multiple ways for players to create and join sessions.

  • Setup - Attached to an empty in the same scene as the Netcode objects, also marked as DontDestroyOnLoad so lobby state persists with the host.
  • Create Lobby - Takes an entered name and a public/private toggle to create the lobby. Relay is configured at this step so players can connect without manual port-forwarding or network setup. The host runs the server; clients connect to it through Relay.
  • Quick Join - Uses Unity Lobby's QuickJoinLobby to find any available session, grabs the relay data from the joined lobby, and starts the client connection through Netcode.
  • Join with Code - Takes an entered lobby code, finds and joins the associated lobby, sets up relay data, then starts the Netcode client.
  • Join with ID - Triggers when a player clicks a specific lobby in the list. Joins by lobby ID, sets up relay data, and starts the Netcode client.
  • List Lobbies - Queries available lobbies, filters out full ones, and updates a public list used to populate the lobby browser UI.
  • Heartbeat & Periodic Refresh - Two timed handlers keep lobbies alive in Unity's service and keep the visible lobby list current.

Implementing three separate join paths (Quick Join, Code, and ID) mirrors how production multiplayer games handle matchmaking UX. Relay removes the need for players to configure networking, which is a real barrier in player-hosted games.

Network Player Prefab

I structured the networked player prefab to group all gameplay, visual, camera, and HUD systems in a clean, intentional hierarchy.

  • Root - An empty that holds all gameplay scripts and acts as the parent for all child systems.
  • Camera Hierarchy - The camera and its controller are parented under a separate empty. This keeps all camera-related logic grouped and separable from the rest of the player.
  • Player Visual - Holds all available models. Only the selected model is toggled on when the scene loads. Each model has its own animator driven by the Player Animation script.
  • Step Ray Lower / Upper - Two empty objects that drive stair-step logic. When only the lower raycast hits geometry (and not the upper), the player is moved upward on the Y axis in the direction of movement, allowing smooth step traversal.
  • HUD - UI elements for health, ammo, and crosshair. Currently includes a delay indicator used to test input latency during development.

A well-organized prefab makes iteration faster, debugging easier, and onboarding clearer. Keeping movement logic, camera logic, visuals, and UI in separate branches means changes to one don't break others.

Network Zombie Prefab

I worked with a zombie prefab where scripts and colliders are kept on a separate empty from the visual model.

  • All gameplay logic and colliders live on a root empty, not on the model itself.
  • The visual model is a child that can be swapped, updated, or animated independently without touching gameplay code.

Separating logic from visuals is a practical prefab design habit. It means artist or animation changes don't risk breaking hitboxes or scripts, and the model can be iterated on without touching gameplay.

Sound FX and Music

I set up the audio architecture for the project, keeping sound effects and music on separate systems with independent volume control.

  • Sound FX Setup - Sound effects play through a dynamically spawned audio source prefab. Spawning is handled entirely through the SoundFXManager. Any system that wants to play a sound calls a function through the manager's instance. All sound effects play in a 3D spatial environment so they have positional falloff.
  • Music Setup - Music plays from a pre-placed empty object already in the scene. It starts on spawn, loops continuously, and plays through a 2D environment so it has no positional effect.
  • Volume - Volume for both channels is handled through the SoundMixerManager. Music and sound effects are controlled independently, so adjusting one doesn't affect the other. A master volume slider can lower everything at once.

Splitting sound effects and music into separate spawning systems and mixer channels is a standard production audio pattern. It gives designers and players independent control over each layer without them interfering with each other.

Sound FX Manager

I built a singleton manager responsible for all runtime sound effect spawning in the game.

  • Implemented as a DontDestroyOnLoad singleton - if additional instances are loaded, they self-destroy to prevent duplicates.
  • PlaySoundFXClip - Takes an audio clip, a world position, a volume value, and a spatial blend. Spawns an audio source prefab at that position, configures it with the provided settings, plays the clip, then destroys the object once playback finishes.
  • PlayRandomSoundFXClip - Takes an array of audio clips along with the same position, volume, and spatial blend parameters. Picks a random clip using the array length as the upper bound, then spawns and plays it the same way as the single-clip version.

Routing all sound effect playback through a single manager makes it easy to add new sounds to any system without duplicating audio logic. The random clip variant adds natural variation to repeated sounds like footsteps or impacts.

Sound Mixer Manager

I built a singleton volume manager that controls audio mixer levels for sound effects, music, and master volume.

  • Implemented as a DontDestroyOnLoad singleton - destroys extra instances if more than one is loaded.
  • Set[Type]Volume - Each function targets a specific mixer group (SFX, Music, or Master) and applies volume using Log10. Unity's audio mixers use an exponential decibel scale, so the raw slider value is converted through Log10 to produce a perceptually linear result for the player.

Using Log10 for volume conversion is a technical detail that matters a lot in practice - a linear slider without it would make the lower half of the range nearly silent and the upper half barely audible as a difference. Getting this right means the volume slider actually feels proportional to the player.

Notes

Gameplay Loop

Players spawn into the map and must quickly find and use weapons to survive incoming zombie attacks. After short intervals, zombies begin running toward the closest player, creating constant pressure and forcing players to balance movement, aim, ammo, and survival decisions. When a zombie is killed, it can randomly drop health or ammo, adding a resource-management layer to the combat loop.

  1. Explore the map and pick up weapons
  2. Fight off approaching zombies
  3. Manage ammo and reload timing
  4. Recover health and supplies from drops
  5. Respawn and re-enter the fight after death
