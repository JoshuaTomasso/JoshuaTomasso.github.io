Overview

Killshot Momentum is a first-person momentum-based game built in Unreal Engine as part of a team. The objective is to complete each map as fast as possible while navigating enemies that block your path and misdirections designed to throw you off course. When you reach the end portal, a UI menu displays your completion time and options to continue.

My contributions focused on UI systems, AI behavior, and the overall game flow — building the menus, settings, and enemy logic that tie the experience together.

  • Full UI system including Main Menu, Pause Menu, Settings, Key Bindings, and Map Complete screen
  • AI enemy behavior using Behavior Trees and AIPerception
  • Game Instance–driven level loading and session management
  • Settings system built on Unreal's Game User Settings with hardware benchmarking

What I Worked On

Settings Menu

I built a full-screen settings UI using Unreal's Game User Settings system, giving players control over display and graphics options.

  • UI Layout - A dark-themed widget where each row shows a setting label with horizontal selector buttons. Options include Window Mode, Display Resolution, Frame Rate Limit, View Distance, Post Processing, Anti-Aliasing, Texture Quality, and Shadow Quality.
  • Window Mode & Resolution - Each window mode button retrieves Game User Settings, applies the correct fullscreen enum, and updates the UI to reflect the active selection. The resolution dropdown splits the selected string into width and height values, constructs an IntPoint, and applies it via Set Screen Resolution.
  • Save & Back - The Apply button saves all current settings to disk via Game User Settings. The Back button creates and adds the previous menu widget to the viewport, then removes the settings screen.
  • Optimal Settings - Runs a hardware benchmark through Game User Settings and automatically loads the recommended values back into the UI, populating all settings fields with hardware-appropriate defaults.

Hooking into Game User Settings means player preferences persist between sessions without any custom save logic. The hardware benchmark feature lets players get a reasonable starting configuration on any machine without manual tuning.

Map Complete Menu

I built the end-of-level screen that displays the player's completion time and handles navigation after finishing a map.

  • UI Layout - Displays a "You Finished!!!" header, a timer showing elapsed time in MM:SS:MS format, and three buttons — Play Again, Next Level, and Main Menu.
  • Timer Display - On construction, the widget casts to the Game Instance and Game Mode to retrieve the elapsed time. The value is split into minutes, seconds, and milliseconds using division and floor operations, then formatted and displayed on screen.
  • Navigation Buttons - Each button calls a corresponding function on the Game Instance — Main Menu loads the main menu level, Play Again reloads the current level, and Next Level loads the next one in sequence.
  • Set Win (Auto Return) - A custom event that hides the button area, switches the active widget to the results display, then sets a one-shot timer. When it expires, it fires a LoadMainMenu event to automatically return the player to the main menu after a set delay.

Routing all level transitions through the Game Instance keeps the navigation logic centralized. The auto-return timer means the game never gets stuck on the results screen even if the player walks away.

Pause Menu

I built the pause menu overlay that handles game state, input mode switching, and navigation to sub-menus.

  • UI Layout - A centered overlay widget with a "Pause Menu" header and five buttons — Resume, Settings, Key Bindings, Restart, and Main Menu.
  • Resume - Unpauses the game, switches input back to game-only mode, hides the cursor, updates the player's pause state flag, and removes the widget from the viewport.
  • Settings / Key Bindings - Creates and adds the corresponding widget to the viewport, then removes the pause menu. Each sub-menu follows the same pattern with a different target widget class.
  • Restart & Main Menu - Call their respective load functions on the Game Instance to reload the current level or return to the main menu.

Handling input mode and cursor state directly in the Resume logic means the game never gets stuck in a state where the cursor is visible during play or locked during menus. All sub-menu navigation follows a consistent swap pattern, making it easy to extend.

Main Menu

I built the title screen that initializes the settings system and routes the player into the game or to supporting screens.

  • UI Layout - A styled title screen featuring the "Killshot Momentum" branding with a character illustration and three buttons — Play Game, Key Bindings, and Quit.
  • Event Construct - On construction, casts to the Game Instance to store a reference, creates and stores a Settings Widget, then calls Load Settings to populate any saved preferences.
  • Key Bindings - Removes the current menu widget and creates the Key Bindings widget, adding it to the viewport with the player controller as the owning player.
  • Play Game - Calls Load First Level on the Game Instance to begin the game.
  • Quit - Calls Quit the Game on the Game Instance to close the application.

Initializing the Settings Widget silently on construction means saved settings are always restored before the player does anything, without requiring an extra step or screen. Centralizing Play and Quit through the Game Instance keeps startup logic consistent with the rest of the flow.

Key Binding Menu

I built a reference screen that displays all game controls and handles navigation back to whichever menu opened it.

  • UI Layout - A three-column reference screen displaying all controls grouped by category — Movement (W/A/S/D, Space Bar, Left Shift), Attack (Left Mouse Click), and Level Keys (P to Pause, F to Reset Level). A Back button sits in the bottom right corner.
  • Back (Main Menu Version) - Removes the current widget and creates the Main Menu widget, adding it to the viewport with the player controller as the owning player.
  • Back (Pause Menu Version) - Follows the same pattern but creates the Pause Menu widget instead, returning the player to the pause screen.

Having two versions of the Back button — one for the main menu context and one for the pause menu context — means the Key Bindings screen works correctly regardless of where it was opened from, without needing to track the previous screen at runtime.

AI Controller & BT Look At

I implemented the enemy AI system using an AI Controller with perception detection and a Behavior Tree that keeps enemies facing the player.

  • AI Controller Overview - Uses an AIPerception component to detect the player and drives behavior through the BT_LookAt behavior tree.
  • Construction Script - Binds the HandlePerception function to the AIPerception component's On Target Perception Updated event so any perception changes are handled automatically.
  • Event On Possess - When the controller possesses a pawn, it immediately starts running the assigned behavior tree asset.
  • Handle Perception - Breaks the incoming AI Stimulus to check if the player was successfully sensed. If true, sets the Player key on the blackboard with the detected actor. If false, clears the Player key, removing the target from the behavior tree.
  • Behavior Tree - A simple tree using a single Sequence node that continuously runs the BTTask_FaceObject task. The blackboard holds references to SelfActor and Player.
  • BTTask_FaceObject - On execute, gets the AI pawn's current rotation and location, finds the look-at rotation toward the player's location, smoothly interpolates to it using RInterp, applies the result via Set Actor Rotation, then finishes the task as a success.

Binding perception handling in the Construction Script keeps the event wiring automatic — no manual setup required when placing enemies in a level. Using RInterp for rotation means the look-at feels smooth rather than snapping instantly, which reads better as a game feel detail.

Notes

Gameplay Loop

Players move through each map as fast as possible using the full movement toolkit — running, jumping, and dashing — while avoiding enemies that block key paths and navigating intentional misdirections built into the level layout. Reaching the end portal triggers the Map Complete screen, which shows the player's elapsed time in MM:SS:MS format and options to replay, advance, or return to the main menu.

Scene / Flow Overview

The game includes a complete navigable flow rather than a single isolated level:

  • Main Menu — title screen with Play Game, Key Bindings, and Quit
  • Key Bindings screen — accessible from both the Main Menu and Pause Menu
  • Settings screen — graphics and display options with persistent save support
  • Gameplay maps — momentum-based traversal with enemies and misdirections
  • Pause Menu — Resume, Settings, Key Bindings, Restart, and Main Menu
  • Map Complete screen — timed results display with Play Again, Next Level, and Main Menu
