Overview

Hidden Fangs is a multiplayer social deduction game inspired by Werewolf, built entirely in Unreal Engine 5 Blueprints. Players spawn in a village and are randomly assigned hidden roles at the start of each match. Gameplay alternates between a Day Phase, where players discuss and vote to eliminate a suspected werewolf, and a Night Phase, where players with special roles use their unique abilities. The objective is for the villagers to identify and eliminate all werewolves before the werewolves eliminate enough villagers to take control of the village.

Social deduction lives or dies on communication, so the project includes both in-game text chat and voice chat, giving players the tools to bluff, accuse, and coordinate in real time.

  • Server-authoritative night action submission and resolution
  • DataTable-driven role assignment and hidden-role reveal
  • Replicated day/night phase management with synced UI timers
  • Real-time werewolf pack coordination for shared night targeting
  • Tested with up to 4 simultaneous PIE instances for multiplayer validation

What I Worked On

Role Assignment & Hidden Role Reveal

I built the role assignment system in BP_GameMode, gated on PostLogin so roles are assigned as players connect rather than all at once.

  • DataTable-Driven Roles — Role data lives in DT_Roles, backed by an E_Roles enum and an S_RoleInfo struct, so adding or rebalancing roles doesn't require touching assignment logic.
  • Private Reveal RPC — Each player's role is sent to them individually via a Client_ReceiveRole RPC, keeping role information hidden from other clients instead of relying on client-side filtering of replicated data.

Keeping role assignment server-side and reveal per-client closes off the obvious cheating vector in a hidden-role game — no client ever receives data about a role it isn't entitled to see.

Day / Night Phase Management

Phase state lives on BP_GameState so it replicates cleanly to every client, with a CurrentPhase variable, timer logic, and a WBP_GamePhaseTimer widget that stays in sync across all players.

Driving phase timing from GameState instead of GameMode was a deliberate fix — GameMode only exists on the server, so any UI or Multicast logic tied to phase changes has to live on GameState to reach clients at all.

Night Action Targeting

I'm building the UI and networking for players to submit their night actions (e.g. werewolf kill target) and see them resolve correctly across every connected client.

  • Server-Authoritative Submission — Night actions are submitted through Server_SubmitNightAction on BP_PlayerState, so the server always has final say over what action was taken.
  • Targeting UI — WBP_NightActionTargeting is built from reusable WBP_ButtonAndText widgets, populated per-role with valid targets.
  • Pack Coordination — Werewolves share visibility into each other's target selection in real time via a RepNotify on NightTarget paired with an OnNightTargetChanged Event Dispatcher, so partner vote displays update live.

submission prevents client-side spoofing of actions, while the RepNotify/dispatcher pair lets teammates coordinate without exposing anything to non-werewolf players.

Cabin Spawning System (Debug)

I tracked down and fixed a spawning bug in the village's cabin placement system.

  • Pure Blueprint nodes were re-executing on every wire read, which was silently defeating the duplicate-index prevention logic meant to stop cabins from spawning on top of each other.
  • A mutation bug in a SET Amount to Spawn node was also causing a For Loop to terminate early, under-spawning cabins.

This one taught me to treat Blueprint's pure-node re-execution as a real gotcha rather than an edge case — anything relying on a pure node's result needs to cache it in a variable first if it's read more than once in the same execution path.

Notes

Gameplay Overview

Each match alternates between a Day Phase — where surviving players discuss and vote to eliminate a suspected werewolf — and a Night Phase, where players with special roles secretly act. Villagers win by eliminating all werewolves; werewolves win by reducing villagers to a number where they can no longer be outvoted.

Networking Architecture

The game is built entirely in Blueprints on top of Unreal's replication system, with no C++ layer. Game-wide state (current phase, timers) lives on BP_GameState, per-player state (role, submitted night action) lives on BP_PlayerState, and server-only logic (role assignment, action resolution) lives on BP_GameMode.

  • Widget creation is gated behind Is Local Controller checks to prevent UI from spawning on the wrong client
  • Multiplayer testing is done with up to 4 simultaneous PIE (Play-In-Editor) instances
  • Hidden information (roles, night targets) is sent via private client RPCs rather than filtered from replicated variables

Architectural Lessons Learned

  • GameMode only exists on the server — any logic that needs to reach clients (Multicast RPCs, phase UI) has to live on GameState instead
  • Pure Blueprint nodes re-execute every time their output is read, which can silently break logic that assumes a single evaluation — cache the result in a SET node if it's read more than once
  • Widget creation needs to be gated with Is Local Controller, or every client ends up spawning UI meant only for the owning player
